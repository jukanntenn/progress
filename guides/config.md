# Configuration

Progress stores application configuration in the database, not in the TOML
file. The TOML file (`config.toml`) is a one-time **seed** plus the provider of
**infrastructure** settings. After the first run, the database is the single
source of truth and ongoing edits happen through the web UI (or the API /
`progress config` CLI), not by editing the file.

## How configuration is split

| Layer | Where it lives | How it is edited |
|---|---|---|
| **Infrastructure** — `data_dir`, `workspace_dir`, db path, server bind/port, schedule cron, log level | TOML file / environment variables | Edit the file (needed before the DB can be opened) |
| **App settings** — `language`, `timezone`, `github`, `analysis`, `markpost`, `report`, `notification.channels`, `proposal_trackers`, `changelog_trackers` | Database: one versioned JSON blob (`app_config` table) | Web UI → *Configuration* page; `POST /api/v1/config` |
| **Repositories & owners** | Database: `repositories` / `github_owners` tables | Web UI → *Repositories* / *Owners* sections; `GET/PUT /api/v1/config/{repos,owners}` |

Infrastructure keys cannot live in the database because they are needed to
open the database in the first place (chicken-and-egg). Everything else lives
in the database.

## The TOML file: seed + infra

On first run with an empty database, the file seeds the database:

- app settings → the `app_config` blob,
- `[[repos]]` → the `repositories` table,
- `[[owners]]` → the `github_owners` table.

From then on the database is authoritative. The file's app-config keys are
**ignored** on subsequent runs (only its infrastructure keys are still read at
startup). This is what removes the old conflict where Ansible rewriting
`config.toml` would clobber changes made through the web UI — Ansible only
writes the seed file, which the application no longer consults for app config
after the first deploy.

Required vs optional still applies to the seed: required items (e.g.
`github.gh_token`) must be present in the file (or via environment variables)
so the first run can validate and seed them. Optional items fall back to
sensible defaults.

## Editing configuration

- **Web UI** — the *Configuration* page renders a form from the JSON Schema
  generated by the server and saves the blob with optimistic locking
  (`version`); if someone else changed the config since you loaded the page,
  saving returns `409` and the page refreshes. The *Repositories* and *Owners*
  sections edit the tables via replace-on-save.
- **Secrets** (`github.gh_token`, Feishu `webhook_url`, email `password`,
  Markpost `url`) are masked (`********`) in every response. A secret is only
  updated when you enter a new value; leaving the mask untouched preserves the
  stored value.
- **CLI**:
  - `progress config import` — re-import the TOML into the database
    (file → DB; overwrites the blob and replaces repos/owners).
  - `progress config export` — dump the database config to TOML (DB → file).

Use these to move configuration between the file and the database intentionally
(for example, seeding a fresh deploy from a checked-in file, or snapshotting
the live DB to a file).

## Precedence

- **Infrastructure**: environment variables > TOML file > defaults (resolved at
  startup, every run).
- **Application config**: the database is the single source of truth. Environment
  variables for app keys are only captured **at seed time** (the first run);
  afterwards the database wins. To force a re-import from the file, run
  `progress config import`.

## Owner Monitoring

`[[owners]]` seeds the `github_owners` table on first run; afterwards manage
owners via the web UI (*Owners* section) or `progress config import`.

### Fields

- `type` (required): `"user"` or `"organization"`
- `name` (required): GitHub username or organization name (cannot be empty)
- `enabled` (optional): `true`/`false` (default `true`) — a disabled owner is
  kept but skipped at check time (not deleted)

### Example (seed file)

```toml
[[owners]]
type = "organization"
name = "bytedance"
enabled = true

[[owners]]
type = "user"
name = "torvalds"
```

## Proposal Tracking

`proposal_trackers` seeds the proposal kinds to track; afterwards manage them
via the web UI or `progress config import`. All repository details (URL,
branch, directory, file patterns) are configured internally — you only specify
which kinds to enable.

### Supported Kinds

| Kind | Repository | Description |
|---|---|---|
| `eip` | ethereum/EIPs | Ethereum Improvement Proposals |
| `erc` | ethereum/ercs | Ethereum Request for Comments |
| `pep` | python/peps | Python Enhancement Proposals |
| `rfc` | rust-lang/rfcs | Rust Request for Comments |
| `dep` | django/deps | Django Enhancement Proposals |

### Example (seed file)

```toml
proposal_trackers = ["eip", "erc", "pep", "rfc", "dep"]
```

## Changelog Tracking

`[[changelog_trackers]]` seeds the `changelog_trackers` list (stored in the
blob); afterwards manage them via the web UI or `progress config import`.

### Fields

- `name` (required): a human-readable tracker name
- `url` (required): changelog URL (HTTP/HTTPS)
- `parser_type` (required): one of `"markdown_heading"`, `"html_chinese_version"`
- `enabled` (optional): `true`/`false` (default `true`)

### Parser Types

- `markdown_heading`: Markdown changelog with headings like `## 1.2.3`; the
  description is the text until the next version heading.
- `html_chinese_version`: HTML changelog with patterns like `uTools v7.5.1`;
  the description is the text until the next version pattern.

### Notification Behavior

- Changelog trackers are checked sequentially on startup.
- The first check always sends a notification (when `last_seen_version` is empty
  in the database) to help validate configuration.
- New version detection uses string comparison
  (`latest_version != last_seen_version`).
- If multiple new versions are found since `last_seen_version`, all are included
  in the generated report.
- When updates exist across trackers, one merged Markpost report is generated
  and a single notification is sent.

### Example (seed file)

```toml
[[changelog_trackers]]
name = "Vite"
url = "https://raw.githubusercontent.com/vitejs/vite/main/packages/vite/CHANGELOG.md"
parser_type = "markdown_heading"
enabled = true
```

### Troubleshooting

- HTTP failures: confirm the URL is reachable from the runtime environment and
  does not require authentication.
- Parse errors: verify `parser_type` matches the actual changelog format and
  that the changelog still contains recognizable version markers.
- Repeated notifications: if notifications fail, `last_seen_version` is not
  updated, so the same version may be retried on the next run.
